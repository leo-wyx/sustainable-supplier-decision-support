# Parameter Assumption Register

## Why a Parameter Register Is Needed

The ESG Supplier Decision Support project spans M1 to M4 plus the management Dashboard, with each layer relying on parameters that govern qualification thresholds, scoring preferences, allocation rules, and resilience constraints. As the project evolves from M1/M2 into M3 allocation and M4 resilience simulation, the number of configurable parameters grows. Without a central register it becomes difficult to distinguish:

- Which parameters are grounded in available project data.
- Which parameters are scenario assumptions made for MVP demonstration.
- What real-world data would be required to replace each assumption.

This register provides that traceability. It is the single source of truth for each parameter's current status, source, and intended replacement path.

## Real Inputs vs. Scenario Assumptions

### Real inputs derived from available supplier data

These fields are computed from or directly read from the current supplier dataset and module outputs. They are not treated as hidden scenario assumptions:

- M1 supplier qualification status: PASS, LIMITED, FAIL / Reserve Pool.
- Supplier base attributes: country, category, annual contract value, certification status, and available operational indicators.
- Existing M2 output fields used for downstream reference, such as cost-related score, ESG score, risk diagnostic, and strategic tier.
- Revised M2 output fields derived from existing M2 scores, including Cost-ESG shortlist status, diagnostic risk level, shortlist stability, and M3 eligibility flag.

This does not mean these fields are perfect real-enterprise data. It means they are part of the current project dataset and are not newly invented inside M3.

### Scenario assumptions for the MVP

These parameters are explicitly marked as assumptions for the current MVP phase and are collected in `config/parameter_config.csv`:

- **M3 demand value** - currently TBD; must come from an external procurement plan, demand forecast, or scenario input in enterprise use.
- **M3 capacity proxy method** - currently uses annual contract value as a proxy when real supplier capacity is unavailable.
- **M3 allocation weights** for cost, ESG, and risk - scenario preferences pending management or procurement policy confirmation.
- **M3 constraints** such as maximum single-supplier share, LIMITED supplier cap, and minimum supplier count - sourcing policy assumptions pending policy confirmation.
- **M2 revised shortlist assumptions** such as category-level shortlist share, minimum category shortlist size, balanced Cost-ESG weights, and stable shortlist scenario count.
- **M2 Scenario Premium Layer assumptions** such as delay cost rate, risk premium schedule, and quality cost rate. These are optional cost-conversion parameters, not M1 qualification rules.

### Pending parameters

Parameters marked `status: pending` in the register have no confirmed value. They are placeholders that must be populated or explicitly assigned through a documented scenario before M3 can produce meaningful allocation results.

## Current MVP Approach

The current MVP uses transparent assumptions where real enterprise data is unavailable. This is by design:

1. **Traceability** - every assumption is flagged with its source in the `current_source` column, so no parameter appears authoritative when it is not.
2. **Replaceability** - the `real_company_replacement` column documents what real data would replace each assumption in an enterprise deployment.
3. **Status visibility** - the `status` column distinguishes `provisional` values from `pending` values.

The design principle is:

> The model structure can be evidence-based even when some parameters remain scenario-based, as long as the scenario parameters are explicitly registered and explained.

## Critical Constraint: M3 Demand Must Remain External

M3 demand (`m3_demand_value`) is a quarterly category demand input measured in USD. It must come from a procurement plan, demand forecast, or external scenario input that is independent of the supplier dataset.

M3 demand must not be generated by applying a fixed ratio such as `DEMAND_RATIO = 0.50` to historical contract values. Doing so would create a circular reference: allocation would re-allocate the same historical spend that was used to derive demand, producing a self-validating result that masks supply gaps and overstates feasibility.

Historical annual contract value may still be used as:

- a supplier scale proxy;
- a capacity proxy when real capacity is unavailable;
- a risk exposure reference;
- a benchmark for whether an allocation scenario looks realistic.

It should not be treated as the source of future quarterly demand.

## Real-Company Replacement Path

Real enterprises deploying this system should replace the scenario assumptions with company-specific data:

| Parameter | Replace with |
|---|---|
| `m3_demand_value` | Quarterly procurement plan, MRP output, sales and operations planning output, or demand forecast |
| `m3_capacity_proxy_method` | Supplier committed capacity, production capacity, available capacity, or approved allocation limit |
| `m3_max_single_supplier_share` | Company supplier concentration policy or category sourcing rule |
| `m3_limited_supplier_cap` | Supplier qualification policy or risk appetite rule |
| `m3_min_supplier_count` | Category sourcing policy, dual-sourcing requirement, or resilience policy |
| `m3_allocation_weight_cost` | Management preference, procurement policy, or category strategy |
| `m3_allocation_weight_esg` | ESG procurement policy, sustainability target, or customer/regulatory requirement |
| `m3_allocation_weight_risk` | Enterprise risk appetite, supplier risk policy, or disruption tolerance |
| `m2_revised_shortlist_share` | Category manager shortlist policy or sourcing-stage design |
| `m2_revised_min_category_shortlist` | Category sourcing policy or buyer-defined shortlist size |
| `m2_revised_stable_scenario_count` | Management preference for shortlist robustness |
| `m2_revised_balanced_cost_weight` | Category strategy or procurement policy |
| `m2_revised_balanced_esg_weight` | ESG sourcing policy, sustainability target, or customer/regulatory requirement |
| `m2_cost_premium_enabled` | Management approval that company-specific premium cost data is available and suitable for use |
| `m2_cost_delay_weight` | Category-specific delay cost policy or contract penalty weighting |
| `m2_delay_cost_rate` | Contract late-delivery penalty clauses, ERP late-delivery history, or documented line-stoppage cost |
| `m2_cost_risk_weight` | Finance or risk-management approved weighting for monetized supplier risk |
| `m2_risk_premium_schedule` | Historical disruption loss data, insurance premium, finance-approved risk rate, or supplier risk-pricing policy |
| `m2_cost_quality_weight` | Category-specific quality cost weighting |
| `m2_quality_cost_rate` | Scrap, rework, return, warranty, rejection, or defect-cost records |

## Scenario Premium Layer Boundary

The Scenario Premium Layer belongs to the M2 Adjusted Cost Index path, not to M1. M1 should continue to decide supplier eligibility through qualification gates and should output risk signals such as `M1_Status` and `M1_Risk_Vector`. Premium components only convert selected risk, delay, or quality signals into cost impact when the company has a defensible replacement data source.

For the current MVP, premium components remain disabled by default. If they are used in a demonstration scenario, the output should label them as scenario-based and should not present them as real enterprise cost. In a company deployment, each premium coefficient must be replaced by contract, ERP, quality, finance, or risk-management data before it is treated as an operational decision parameter.

## How This Supports M3

The M3 extension allocates category demand across qualified suppliers using a lightweight, constraint-aware allocation prototype. The parameters registered here serve as:

- **Hard constraints**: `m3_max_single_supplier_share`, `m3_limited_supplier_cap`, and `m3_min_supplier_count` define the feasible allocation space.
- **Objective or preference weights**: `m3_allocation_weight_cost`, `m3_allocation_weight_esg`, and `m3_allocation_weight_risk` determine trade-offs in the allocation preference score.
- **Input data method**: `m3_capacity_proxy_method` defines how supplier capacity is estimated when real data is unavailable.
- **External boundary**: `m3_demand_value` is the total category demand to be allocated and is treated as external input.

The initial M3 implementation may read these parameters at runtime or reference them during allocation scenario design. Parameters marked `pending` should not be silently replaced with hidden defaults. If a value is required for an MVP run, it should be assigned through a documented scenario row or clearly marked as a temporary assumption.

M3 lightweight allocation has been implemented as an extension layer. Its demand, capacity, and policy parameters remain illustrative assumptions and should be replaced by real enterprise procurement, capacity, and sourcing-policy data in deployment.
